Fichier angular.json


Le fichier angular.json est un élément central dans un projet Angular.
Il contient les configurations nécessaires pour gérer les projets, les builds, les schémas, et bien d'autres paramètres.

Structure et Contenu


code :

              {
                  "$schema": "./node_modules/@angular/cli/lib/config/schema.json",
                  "version": 1,
                  "newProjectRoot": "projects",
                  "projects": {
                    "mini-projet-signals": {
                      "projectType": "application",
                      "schematics": {},
                      "root": "",
                          ..... 
                          }
                        }
                      }
  • "$schema" : Indique le chemin vers le fichier de schéma qui valide et guide la structure du fichier angular.json.
  • "version" : Spécifie la version du format utilisé pour ce fichier de configuration (ne pas utilisable par le développeur).
  • "newProjectRoot" : Définit le répertoire racine (projects) où les nouveaux projets Angular seront créés. ( on peut modifier cette valeur pour changer l'emplacement des nouveaux projets. )

                "$schema": "./node_modules/@angular/cli/lib/config/schema.json",
                "version": 1,
                "newProjectRoot": "projects",
                "projects": {
                  "mini-projet-signals": {
                    "projectType": "application",
                    "schematics": {},
                    "root": "",
                    "sourceRoot": "src",
                    "prefix": "app",
                    "architect": {
                      "build": {
                        "builder": "$#64;angular-devkit/build-angular:application",
                       ..... 
                         }
                        }
                      }
                    }
  • "projects" : La section projects contient tous vos projets Angular (au cas où vous travailleriez avec plusieurs applications ou bibliothèques). Chaque projet aura sa propre configuration.
  • "projectType" : Indique le type de projet, généralement "application" pour une application Angular ou "library" pour une bibliothèque.
  • "schematics" : Configure les générateurs de code (comme les composants, services, etc.) pour appliquer des options ou des comportements par défaut lors de leur création.
  • "root" : Dpécifie le chemin racine du projet par rapport au workspace, généralement vide ("") pour les projets situés à la racine. ( ce champ est automatiquement défini lors de la création du projet et ne nécessite pas de modification manuelle. )
  • "sourceRoot" : Définit le chemin vers le répertoire contenant les fichiers source du projet (par défaut, src).
  • "prefix" : Définit le préfixe utilisé pour les balises HTML des composants Angular.
  • "architect" : Contient les configurations pour les différentes cibles (build, test, lint, etc.) du projet Angular.
  • "build" : Céfinit les options et les configurations pour le processus de build de l'application (par exemple, les paramètres de production, le minification, etc.).
  • "builder" : spécifie le nom du builder utilisé pour le processus de build ,(généralement @angular-devkit/build-angular:application ou similaire comme (@angular-builders/custom-webpack:browser)).

  • "options" : contient les configurations spécifiques au processus de build d'une application Angular, comme les chemins, les fichiers sources, les styles, et les options pour le serveur.
  • "outputPath" : spécifie le chemin où les fichiers de production (dist) seront générés.
  • "index" : indique le chemin vers le fichier index.html utilisé comme point d'entrée pour l'application.
  • "browser" : spécifie le fichier TypeScript utilisé comme point d'entrée principal pour le navigateur.
  • "polyfills" : liste les bibliothèques ou modules qui doivent être chargés en premier pour assurer la compatibilité avec les navigateurs anciens.
  • "tsConfig" : spécifie le chemin vers le fichier tsconfig utilisé pour le projet, définissant les options de compilation TypeScript.
  • "assets" : liste les fichiers et répertoires à inclure dans le build, comme les favicon, les images, et d'autres ressources statiques.
  • "glob" : spécifie un modèle de chemin pour inclure des fichiers lors du build (par exemple, **/* pour tout fichier).
  • "input" : indique le chemin d'entrée des fichiers que l'on souhaite inclure dans le build (par exemple, public pour le dossier public).
  • "styles" : liste les feuilles de styles CSS à inclure dans le build de l'application.
  • "scripts" : liste les fichiers JavaScript à inclure dans le build, comme des bibliothèques ou des scripts supplémentaires.
  • "server" : indique le fichier TypeScript utilisé comme point d'entrée pour le serveur, souvent utilisé dans des configurations SSR (server-side rendering).
  • "outputMode" : définit le mode de sortie pour le build (par exemple, "server" pour les configurations SSR).
  • "ssr" : contient les configurations spécifiques pour le serveur-side rendering (SSR), comme l'entrée du serveur.
  • "entry" : spécifie le fichier TypeScript utilisé comme point d'entrée pour le serveur dans le cadre du SSR.

            {
                   ...
                    "architect": {
                      "build": {
                        "builder": "$#64;angular-devkit/build-angular:application",
                        "options": {
                          ...
                          },
                          "configurations": {
                            "production": {
                              "budgets": [
                              {
                                  "type": "initial",
                                  "maximumWarning": "500kB",
                                  "maximumError": "1MB"
                                  },
                                {
                                  "type": "anyComponentStyle",
                                  "maximumWarning": "4kB",
                                  "maximumError": "8kB"
                               }
                              ],
                              "outputHashing": "all"
                              },
                            "development": {
                              "optimization": false,
                              "extractLicenses": false,
                              "sourceMap": true 
                              }
                            },
                          "defaultConfiguration": "production"
                       ..... 
                         }
                        }
                      }
                    }
  • "configurations" : définit les options spécifiques pour le build en production et en développement, telles que les budgets de taille, le hachage des fichiers, et les optimisations.
  • "production" : contient les paramètres spécifiques pour le build en mode production, comme les budgets de taille et le hachage des fichiers.
  • "budgets" : définit les limites de taille pour le code initial et les styles, en générant des avertissements ou des erreurs si les seuils sont dépassés.
  • "type" : spécifie le type de ressource pour laquelle le budget est appliqué, par exemple initial pour le code initial ou anyComponentStyle pour les styles.
  • "maximumWarning" : définit la taille limite pour laquelle un avertissement sera déclenché si dépassée.
  • "maximumError" : définit la taille limite au-delà de laquelle une erreur sera déclenchée, empêchant le build de passer en production si elle est dépassée.
  • "outputHashing" : définit le mode de hachage des fichiers de sortie (comme all, media, ou bundles) pour forcer des noms uniques des fichiers lors de chaque build.
  • "development" : contient les paramètres spécifiques pour le build en mode développement, comme la désactivation de l'optimisation, l'extraction des licences, et l'activation des sourcemaps.
  • "optimization" : indique si les optimisations (comme le minification, le tree-shaking) doivent être activées ou non en mode développement.
  • "extractLicenses" : indique si les licences des dépendances doivent être extraites et ajoutées aux builds en mode développement.
  • "sourceMap" : indique si les fichiers source map doivent être générés pour le débogage en mode développement.
  • "defaultConfiguration" : définit la configuration par défaut à utiliser pour le build (par exemple, production ou development).
